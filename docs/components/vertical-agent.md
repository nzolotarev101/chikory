# Component: Vertical Agent (Stage 3)

**Phase**: P5 (months 9–18) · **WPs**: WP-501..507 · **Requirements**: ST-5 · **Spec**: §10 Stage 3
**Resolution note**: decision *framework* and architecture are fixed now; the vertical *choice* is data-gated on Stage 2 (this is by design, not an omission).

## 1. Purpose

Pick one full-app vertical underserved by vibe-coders and ship a **templated agent** that generates and *maintains* that app class with measurable SLOs. Stage 3 proves the thesis commercially: not "a platform you could use," but "a result you can buy."

## 2. Vertical selection framework (WP-501)

Candidates from spec §10: internal admin tools · B2B SaaS dashboards · regulated/audit-trail industries.

Scoring rubric (each 1–5, weighted; filled with Stage-2 data, not opinion):

| Criterion | Weight | Data source |
|---|---|---|
| Observed demand: what are Stage-2 customers actually running? | 3× | WP-306 dataset task taxonomy |
| Verifiability: can acceptance criteria be machine-checked end-to-end? | 3× | benchmark experience — verticals with checkable SLOs favored |
| Brownfield share: maintenance revenue, not one-shot generation | 2× | spec thesis (long-horizon is the wedge) |
| Incumbent absence: vibe-coders weak here? | 2× | market scan refresh |
| Compliance leverage: does our audit-trail/judge story uniquely fit? | 2× | regulated verticals score high *if* WP-406 shipped well |
| Sales reachability with current team | 1× | pipeline evidence |

Decision = ADR-004 (written at P5 start). Strong prior from current evidence: **internal admin tools** (highest verifiability, lowest compliance barrier to start, natural brownfield tail — every generated tool needs maintenance, which is recurring usage of the platform).

## 3. Architecture: template = constrained TaskSpec factory, not new runtime

No new execution machinery. A vertical template is a **pack** layered on the existing control plane:

```
vertical-pack/
  blueprint/        opinionated app skeleton (stack pinned: e.g. Next.js + Postgres + auth)
  criteria/         reusable AcceptanceCriterion sets with `check` commands (the SLO library)
  rubrics/          judge rubric packs tuned to the app class (extends WP-215 mechanism)
  playbooks/        TaskSpec generators: "new app", "add entity/CRUD", "add report",
                    "migrate schema", "upgrade deps", "fix from bug report"
  slo.yaml          the measurable promises (below) + their verification commands
```

- Playbooks emit ordinary `task.yaml` → ordinary runs → ordinary journals. Everything (judge gates, durability, pricing, trace browser) is inherited; the pack adds domain opinion only. This keeps NF-1 honest at Stage 3 — still no framework.
- Maintenance mode: a long-lived "steward" schedule per deployed app — dependency upgrades, bug-report-to-fix runs (brownfield continuity via WP-204 cross-session archival memory).

## 4. SLOs (the Stage-3 differentiator — measurable, contractual)

Defined per pack in `slo.yaml`; examples for admin-tools pack:
- New CRUD entity request → deployed behind review gate in ≤ N hours, all pack criteria green.
- 100% of changes carry a judge-verified audit entry (diff + tests + verdict) — the compliance sell.
- Dependency CVE (high) → patched + verified ≤ 72h.
- Regression rate: < X% of steward changes rolled back per quarter (measured from journals — the metric is native).

SLO measurement = JIF queries; reports generated by trace-service. **The audit trail per change is itself the product for regulated buyers.**

## 6. Entry / exit

**Entry**: Stage-2 exit criteria met (control-plane.md §8) + dataset large enough to score the rubric honestly.
**Exit (the enterprise-product bar)**: ≥1 vertical pack GA with contractual SLOs; design partners renewing; steward runs are the dominant (recurring) usage; sales motion repeatable without founder-led demos.
